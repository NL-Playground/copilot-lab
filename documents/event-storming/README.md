# 事件風暴文件目錄

## 用途
存放專案的事件風暴（Event Storming）相關文件，包含：
- 流程圖截圖
- 業務流程描述
- 領域事件定義
- 系統互動設計

## 檔案格式
- **圖片**: `*.png`, `*.jpg`, `*.svg`
- **說明**: `*.md` (Markdown 文件)

## 已有文件

- `logging.png` - 登入流程圖

## 檔案組織建議

```
event-storming/
├── README.md                    # 本檔案
├── overview.md                  # 整體流程概述

├── login-flow.md               # 登入流程說明
├── user-registration-flow.png  # 註冊流程圖
├── user-registration-flow.md   # 註冊流程說明
└── domain-events.md            # 領域事件定義
```

## Event Storming 組成元素

### 1. Actor (參與者)
系統的使用者或外部系統
- 顏色：黃色
- 範例：網頁終端使用者、管理員

### 2. Command (命令)
觸發事件的操作
- 顏色：藍色
- 範例：登入、建立訂單、更新資料

### 3. Aggregate (聚合)
執行商業邏輯的核心物件
- 顏色：紫色
- 範例：驗證登入、處理訂單、計算金額

### 4. Domain Event (領域事件)
系統中發生的重要事件
- 顏色：橘色
- 範例：使用者已登入、訂單已建立、付款已完成

### 5. Policy (策略)
事件觸發的自動化行為
- 顏色：粉紅色
- 範例：建立登錄日誌、發送通知、更新庫存

### 6. Hotspot (熱點)
需要特別關注的複雜流程或風險點
- 顏色：紅色
- 範例：驗證失敗流程、並發處理、資料一致性

### 7. External Service (外部服務)
系統依賴的外部資源
- 顏色：灰色
- 範例：Redis、資料庫、第三方 API

### 8. Read Model (查詢模型)
CQRS 模式中的查詢視圖
- 顏色：綠色
- 範例：登入狀態、使用者資訊、報表資料

## 參考時機

在以下情況參考此目錄的文件：
- 實作新功能前理解業務流程
- 設計系統架構
- 規劃領域事件
- 理解系統互動關係
- 除錯業務邏輯問題

## 建立新流程圖

1. **繪製流程圖**
   - 使用 Miro、Mural、或實體便利貼
   - 依照 Event Storming 顏色規範

2. **截圖保存**
   - 清晰的高解析度圖片
   - 檔名使用 `{功能名稱}-flow.png`

3. **撰寫說明文件**
   - 對應的 Markdown 檔案
   - 詳細描述流程、決策點、特殊處理

4. **更新 overview.md**
   - 加入新流程的連結和簡介

## 維護原則

1. **流程變更時同步更新**
2. **保持圖文一致性**
3. **標註最後更新日期**
4. **記錄重要的設計決策**
