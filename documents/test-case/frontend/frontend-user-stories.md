## US-FE-001: 使用者登入表單

**作為** 網站訪客  
**我想要** 在登入頁面輸入帳號和密碼  
**以便** 登入系統並存取功能

### 驗收標準

- [ ] 登入頁面顯示帳號輸入框
- [ ] 登入頁面顯示密碼輸入框
- [ ] 密碼輸入框顯示為遮罩（不顯示明文）
- [ ] 登入頁面顯示「登入」按鈕
- [ ] 輸入框有適當的 placeholder 提示文字
- [ ] 表單使用語意化 HTML（label、input type 正確）
- [ ] 支援鍵盤操作（Tab 切換、Enter 送出）
- [ ] 表單欄位有適當的 autocomplete 屬性

### 技術規格

- **組件**: `LoginForm.vue`
- **使用技術**: Vue 3 Composition API
- **樣式**: CSS / Scoped Style
- **相依項目**: 無

### 測試案例

- TC-FE-001: 登入表單渲染測試
- TC-FE-002: 表單欄位輸入測試
- TC-FE-003: 鍵盤操作測試

### 優先級

**High** - 核心功能，必須優先實作

### 預估工時

**4 小時**

### 備註

- 設計需符合 WCAG 2.1 AA 無障礙標準
- 支援 RWD 響應式設計（手機、平板、桌面）
- 預留未來「記住我」功能的擴展空間

## US-FE-002: 登入表單驗證

**作為** 網站訪客  
**我想要** 在送出登入表單前進行前端驗證  
**以便** 即時得知輸入錯誤，減少不必要的 API 請求

### 驗收標準

- [ ] 帳號欄位為空時顯示錯誤訊息「請輸入帳號」
- [ ] 密碼欄位為空時顯示錯誤訊息「請輸入密碼」
- [ ] 帳號欄位失焦（blur）時進行即時驗證
- [ ] 密碼欄位失焦（blur）時進行即時驗證
- [ ] 驗證失敗時禁用「登入」按鈕
- [ ] 錯誤訊息顯示在對應欄位下方，使用紅色文字
- [ ] 錯誤訊息有適當的 ARIA 標籤（accessibility）
- [ ] 驗證通過後「登入」按鈕可點擊

### 技術規格

- **組件**: `LoginForm.vue`
- **驗證邏輯**: Composition API `ref`, `computed`
- **錯誤狀態管理**: Reactive state
- **相依項目**: US-FE-001

### 測試案例

- TC-FE-004: 空白欄位驗證測試
- TC-FE-005: 即時驗證測試
- TC-FE-006: 錯誤訊息顯示測試
- TC-FE-007: 按鈕狀態切換測試

### 優先級

**High** - 提升使用者體驗，減少伺服器負載

### 預估工時

**3 小時**

### 備註

- 驗證規則需與後端 API 驗證規則一致
- 錯誤訊息應支援多語系（i18n 預留）
- 考慮未來擴展更複雜的驗證規則（如密碼強度）

## US-FE-003: 登入 API 請求與回應處理

**作為** 網站訪客  
**我想要** 送出登入表單後看到載入狀態和結果  
**以便** 知道系統正在處理我的請求並得知登入是否成功

### 驗收標準

- [ ] 點擊「登入」按鈕後顯示載入中狀態（Loading spinner 或按鈕文字改為「登入中...」）
- [ ] 載入中時禁用表單輸入和按鈕（防止重複送出）
- [ ] 成功登入後顯示成功訊息「登入成功！」（綠色提示）
- [ ] 成功登入後顯示使用者名稱「歡迎，{username}！」
- [ ] 登入失敗後顯示錯誤訊息（紅色提示）
- [ ] API 請求失敗（網路錯誤）時顯示「無法連線至伺服器，請稍後再試」
- [ ] 所有訊息自動在 3 秒後消失（或提供關閉按鈕）
- [ ] 成功登入後清空密碼欄位（安全考量）

### 技術規格

- **組件**: `LoginForm.vue`
- **API 端點**: `POST /api/login`
- **狀態管理**: `loading`, `error`, `success` reactive states
- **HTTP 客戶端**: Fetch API 或 Axios（規劃中）
- **相依項目**: US-FE-001, US-FE-002

### 測試案例

- TC-FE-008: 登入 API 成功回應測試
- TC-FE-009: 登入 API 失敗回應測試（401）
- TC-FE-010: 登入 API 網路錯誤測試
- TC-FE-011: 載入狀態顯示測試
- TC-FE-012: 訊息自動消失測試

### 優先級

**High** - 核心功能，實現前後端整合

### 預估工時

**6 小時**

### 備註

- 考慮使用 Toast 通知組件顯示訊息（未來可抽取為共用組件）
- 需要處理不同 HTTP 狀態碼（400, 401, 500 等）
- 預留未來 Token 管理的擴展空間（儲存 JWT）

## US-FE-004: 登入錯誤訊息顯示

**作為** 網站訪客  
**我想要** 看到清楚的錯誤訊息說明  
**以便** 了解登入失敗的原因並進行適當的處理

### 驗收標準

- [ ] 帳號或密碼錯誤時顯示「帳號或密碼錯誤，請重新輸入」
- [ ] 帳號被鎖定時顯示「帳號已被鎖定，請於 XX 分鐘後再試或聯絡管理員」
- [ ] 帳號未啟用時顯示「帳號尚未啟用，請檢查您的電子郵件」
- [ ] 伺服器錯誤時顯示「系統發生錯誤，請稍後再試」
- [ ] 錯誤訊息有適當的圖示（警告、錯誤）
- [ ] 錯誤訊息顏色醒目（紅色或橘色）
- [ ] 錯誤訊息有動畫效果（淡入）
- [ ] 提供「重試」或「返回」按鈕（依錯誤類型）

### 技術規格

- **組件**: `LoginForm.vue` 或獨立 `ErrorMessage.vue`
- **錯誤碼對應**:
  - 401 + message: "帳號或密碼錯誤" → 顯示錯誤訊息
  - 423 (Locked) → 顯示鎖定訊息
  - 403 (Inactive) → 顯示未啟用訊息
  - 500 → 顯示系統錯誤訊息
- **相依項目**: US-FE-003

### 測試案例

- TC-FE-013: 錯誤訊息對應測試（各種錯誤碼）
- TC-FE-014: 錯誤訊息樣式測試
- TC-FE-015: 錯誤訊息動畫測試
- TC-FE-016: 多語系錯誤訊息測試（未來）

### 優先級

**High** - 提升使用者體驗，清楚傳達錯誤原因

### 預估工時

**4 小時**

### 備註

- 錯誤訊息需符合 WCAG 無障礙標準（screen reader 支援）
- 考慮建立錯誤訊息對照表（error code mapping）
- 未來可擴展為全域錯誤處理機制

## US-FE-005: 登入成功後顯示使用者資訊

**作為** 已登入的使用者  
**我想要** 在登入成功後看到我的使用者資訊  
**以便** 確認登入成功並了解當前登入狀態

### 驗收標準

- [ ] 登入成功後顯示「歡迎，{username}！」訊息
- [ ] 顯示使用者名稱在頁面右上角或導覽列
- [ ] 使用者名稱旁顯示使用者頭像（如有，否則顯示預設圖示）
- [ ] 登入表單隱藏或替換為使用者資訊面板
- [ ] 提供「登出」按鈕
- [ ] 使用者資訊面板有適當的視覺回饋（動畫、過渡效果）
- [ ] 重新整理頁面後，使用者資訊仍然顯示（Session 持久化）
- [ ] 支援響應式設計（手機版顯示縮略資訊）

### 技術規格

- **組件**: 新增 `UserProfile.vue` 或修改 `App.vue`
- **狀態管理**: 使用 `localStorage` 或 Pinia (規劃中) 儲存使用者資訊
- **顯示資訊**:
  - username (必要)
  - displayName (選用)
  - avatarUrl (選用)
- **相依項目**: US-FE-003

### 測試案例

- TC-FE-017: 使用者資訊顯示測試
- TC-FE-018: 使用者資訊持久化測試（重新整理）
- TC-FE-019: 登出按鈕功能測試
- TC-FE-020: 響應式設計測試

### 優先級

**Medium** - 提升使用者體驗，確認登入狀態

### 預估工時

**5 小時**

### 備註

- 需要設計使用者資訊面板的 UI（與 UI/UX 協作）
- 預留未來擴展功能的空間（個人設定、通知等）
- 考慮使用 Token 管理（JWT 解析取得使用者資訊）
